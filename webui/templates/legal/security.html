Security Policy

Effective: March 8, 2026 • Version 1.0.0

🛡️ SECURITY COMMITMENT

CogniWatch is committed to maintaining the security of our software and protecting user data. This Security Policy outlines our security practices, responsibilities, and how to report security vulnerabilities.

1. SECURITY OVERVIEW

1.1 Security by Design

CogniWatch is built with security as a core principle:

  • Local-first architecture: All scan data remains on your infrastructure
  • Minimal data collection: We collect only essential information
  • Defense in depth: Multiple security layers protect against threats
  • Regular updates: Security patches are released promptly

1.2 Security Model

CogniWatch operates on a trust-but-verify model:

  • Authentication is required for dashboard access
  • API endpoints enforce access controls
  • All inputs are validated and sanitized
  • Rate limiting prevents abuse

2. SECURITY FEATURES

2.1 Authentication & Authorization

  • Token-based authentication: JWT tokens for secure session management
  • Role-based access control: Admin and viewer roles
  • API key support: Programmatic access with scoped permissions
  • Session expiration: Sessions expire after configurable time periods

2.2 Input Validation

  • Parameter validation: All inputs validated against expected formats
  • SQL injection prevention: Parameterized queries only
  • XSS prevention: Output encoding and Content Security Policy
  • CSRF protection: CSRF tokens for state-changing operations

2.3 Network Security

  • HTTPS recommended: Encrypt traffic with TLS
  • CORS configuration: Restrict cross-origin requests
  • IP allowlisting: Optional restriction to trusted IPs
  • Rate limiting: Prevent brute-force and DoS attacks

2.4 Data Protection

  • Encrypted storage: Sensitive data encrypted at rest
  • Secure credentials: Secrets stored in environment variables
  • No data exfiltration: Scan results never sent to external servers
  • Database security: SQLite with restricted file permissions

3. SECURITY CONFIGURATION

3.1 Required Configuration

Before deploying CogniWatch, configure the following security settings:

  • COGNIWATCH_SECRET_KEY - Unique secret key for session encryption
  • COGNIWATCH_ADMIN_TOKEN - Strong token for admin authentication
  • COGNIWATCH_ALLOWED_IPS - Comma-separated list of allowed client IPs (optional)

3.2 Recommended Configuration

  • Enable HTTPS: Use reverse proxy with TLS termination
  • Set up rate limiting: Configure requests per minute
  • Regular password rotation: Change admin tokens periodically
  • Enable logging: Monitor access and errors
  • Backup database: Regular backups of cogniwatch.db

3.3 Production Deployment

For production environments:

  • Run with --debug=false
  • Use a reverse proxy (nginx, Apache)
  • Configure firewall rules to restrict access
  • Enable security headers (HSTS, CSP, X-Frame-Options)
  • Run as non-root user with minimal privileges

4. VULNERABILITY DISCLOSURE

4.1 Responsible Disclosure

We encourage responsible disclosure of security vulnerabilities:

  1. Report privately: Email security@cogniwatch.dev
  2. Include details: Steps to reproduce, impact, affected versions
  3. Allow time for response: We'll acknowledge within 48 hours
  4. Coordinate disclosure: Work with us on public disclosure timeline

4.2 What to Report

We're interested in hearing about:

  • Authentication bypasses
  • SQL injection vulnerabilities
  • Cross-site scripting (XSS)
  • Cross-site request forgery (CSRF)
  • Information disclosure
  • Remote code execution
  • Privilege escalation
  • Denial of service vulnerabilities

4.3 What NOT to Report

Please don't report:

  • Issues requiring unrealistic attack scenarios
  • Theoretical vulnerabilities without proof of concept
  • Issues already fixed in latest version
  • Login CSRF on logout endpoints
  • Social engineering attacks against our team

4.4 Recognition

Security researchers who report valid vulnerabilities will be acknowledged in our security advisory (unless they prefer anonymity).

5. SECURITY UPDATES

5.1 Update Policy

  • Critical patches: Released within 72 hours of confirmed vulnerability
  • Security advisories: Published with each security update
  • Changelog: Security fixes documented in release notes
  • Backward compatibility: Security updates maintain compatibility where possible

5.2 Staying Updated

  • Subscribe to our security mailing list
  • Monitor GitHub releases
  • Check security advisories regularly
  • Enable auto-updates in production

6. INCIDENT RESPONSE

6.1 If You Suspect a Breach

  1. Stop the scan: Halt CogniWatch processes
  2. Preserve evidence: Save logs and database state
  3. Change credentials: Rotate all tokens and API keys
  4. Contact us: Email security@cogniwatch.dev
  5. Review access: Check for unauthorized access

6.2 Our Response

Upon notification of a security incident, we will:

  • Investigate and assess the situation
  • Provide guidance on containment
  • Release patches if needed
  • Communicate findings (as appropriate)

7. COMPLIANCE

7.1 Security Standards

CogniWatch development follows security best practices aligned with:

  • OWASP Top 10 awareness
  • NIST Cybersecurity Framework principles
  • CIS Controls for application security

7.2 Audit Support

Documentation and logs are available to support security audits of your CogniWatch deployment.

8. SECURITY LOGGING

CogniWatch logs security-relevant events including:

  • Login attempts (successful and failed)
  • API access
  • Configuration changes
  • Scanner activity
  • Error conditions

Logs are stored locally and should be reviewed regularly.

9. THIRD-PARTY DEPENDENCIES

CogniWatch uses third-party libraries and services:

  • Flask: Web framework
  • JWT: Token authentication
  • SQLite: Database

We monitor dependencies for security updates and patch promptly.
